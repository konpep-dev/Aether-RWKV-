About

What is Aether?

Aether is a conversational language model built on the RWKV v4 architecture (Receptance Weighted Key Value). Unlike traditional Transformers, RWKV uses a linear attention mechanism that enables efficient CPU inference with no quadratic memory costs.


Trained on a custom dataset of ~22,000 conversations spanning computer science, casual chat, creative writing, advice, and more. Designed to run on modest hardware — a 4-core Intel i5-4570 with 16 GB RAM is sufficient for both training and inference.

Architecture Deep Dive

What is RWKV?

RWKV is a custom RNN architecture that manages to train in parallel like a Transformer while running in constant space like a classic RNN. The name comes from its four core parameters: Receptance, W (time-decay weight), Key, and Value.

Two modes, one model

At training time RWKV is parallelized just like a Transformer. At inference time it is a pure RNN. The same weights, two different compute patterns. This is something Transformers and traditional RNNs both struggle to do.

O(T) instead of O(T²)

Self-attention looks at every previous token, so its cost grows quadratically with sequence length. RWKV only ever updates a fixed-size state vector, so the cost is linear — and constant in memory.

Each channel has its own memory

Inside the state, every channel "forgets" at its own learned rate. Some channels track what's coming next in the sentence; others hold the topic of the paragraph. The model decides which is which.

Built for tiny hardware

Most of RWKV's inference is matrix-vector multiplies — embarrassingly fast on any modern CPU. Aether takes this further by also being trainable on modest hardware in a few hours.

A simplified forward pass

What actually happens inside a single RWKV block, in 8 lines.

# Simplified Aether forward pass — one RWKV block
xx = shift(x) # previous token's hidden state
xk = blend(x, xx, time_mix_k) # mix current+prev into a key
xv = blend(x, xx, time_mix_v) # ...and into a value
xr = blend(x, xx, time_mix_r) # ...and into a receptance
k, v, r = key(xk), value(xv), receptance(xr) # learnable projections
wkv = wkv_update(k, v, decay, bonus) # the O(T) attention step
y = output(sigmoid(r) * wkv) # gate by receptance, project out
x = x + y # residual connection

The single most important idea behind RWKV: the time-decay parameter does not depend on the input. That one design choice is what lets the layer train in parallel (you can precompute all the decay multipliers ahead of time) while still being a real RNN at inference — the entire history is collapsed into a fixed-size state vector.

The math, step by step

Plain-language intuition for each equation, in the order they fire during one forward pass.

1. Time-decay — each channel "forgets" at its own rate

Think of the hidden state as a row of buckets, one per channel. Every step, each bucket slowly leaks away. The decay rate is learned per-channel: a "tense-verb" channel might decay fast (you only need the past few tokens), while a "topic" channel decays slowly (it should stick around for the whole paragraph). The double exponential is a trick to keep the value strictly in (0, 1) no matter what the optimizer tries to set it to.

2. Bonus weight — "this token is special"

The decay rule above treats the current token the same as any other. But in language, the most recent word usually matters most. So we add a small bonus weight that is applied to the current token's contribution only. It's learned too — a tiny asymmetric boost that costs almost nothing.

3. Token-shift — bleed a little of the past in

Before we use the current token, we blend it with the previous token's hidden state. The mixing coefficient is learned per channel. Fast-mixing channels pick up context from one step back; slow-mixing ones stay close to the current token. It's the same trick Apple's AFT (Attention-Free Transformer) popularized — and it works really well.

4. WKV — the heart of the layer

The output is a ratio. The numerator holds "weighted values from all previous tokens, plus this one". The denominator holds the same thing without the values, acting as a softmax normalizer. The u bonus is added only to the current token — past tokens keep their regular weights.

5. State recurrence — two vectors, not one

The "state" is actually two fixed-size vectors: a numerator and a denominator. They are updated in parallel by two near-identical recurrences. Both decay with w, and both accumulate the current token's key — but only the numerator absorbs the value. After a million tokens, you still only need to carry these two vectors forward — no KV cache, no attention matrix.

6. Output gate — "should I update or not?"

Just like in a Transformer where the attention output gets multiplied by V, in RWKV the WKV result gets multiplied by a sigmoid of the receptance. R is the model's "yes/no" — for each channel, it decides whether the new info should actually flow through, or whether the channel wants to ignore this token.

Training

How it was trained

Trained on CPU over ~15 hours with carefully curated data.

Dataset

22,010 conversations across 14 thematic sections:

  • CS knowledge (hardware, networking, OS, algorithms, security, AI)
  • Casual chat & small talk (hobbies, opinions, life advice)
  • Creative writing & storytelling
  • Multi-turn conversations with follow-ups
  • Combinatorial templates for broad coverage

Hyperparameters

  • Batch size: 8 (optimal for CPU)
  • Learning rate: 0.001 → 1e-5 (cosine annealing)
  • Optimizer: AdamW (selective weight decay)
  • Gradient clipping: 1.0
  • Epochs: 12-15 (~15h budget)
  • Checkpoint every 30 min (auto-resume)
Use Cases

Where Aether fits in

Not a 7B model — a 25M specialist. Here's where it makes sense.

Edge & embedded devices

Raspberry Pi, mobile phones, IoT boards. RWKV's constant-space inference means Aether can run on hardware where Transformers simply won't fit. A single state vector, not a growing KV cache.

Privacy-first local assistants

Run a chatbot entirely offline. No API calls, no data leaving the device. Useful for personal journaling, drafting, brainstorming, or any task where conversations should stay private.

Learning & research

Small enough to train end-to-end in hours on consumer hardware, big enough to demonstrate real learning. Great playground for experimenting with initialization, schedulers, datasets, and architectural variants.

Domain-specific fine-tuning

24.7M params is a perfect base for domain adaptation. Fine-tune on a few thousand examples of medical, legal, or company-specific text and get a specialized expert model in minutes.

High-throughput batch jobs

When you need to process millions of documents (classification, extraction, summarization) the per-token cost matters. CPU + RWKV beats GPU + Transformer for low-latency single-token workloads.

Autocomplete & inline assistance

Sub-100ms generation on modest hardware. Drop it into an editor, IDE, or search box to power lightweight completions without a network round-trip.

Scaling Up

How to make Aether bigger

Aether is intentionally small (24.7M) so it trains on a 4-core i5 in hours. The same code scales to much larger models — here is exactly what to change.

1. Architecture config in train.py

# === Aether (current) — 24.7M params, CPU-friendly ===
HIDDEN_SIZE = 480 # 12 layers deep, narrow
NUM_LAYERS = 12
MAX_LENGTH = 64 # context window
DROPOUT = 0.1
BATCH_SIZE = 8 # optimal for CPU
EPOCHS = 20
LEARNING_RATE = 0.001

# === Aether-Medium — ~100M params, still CPU-trainable in ~3-5 days ===
HIDDEN_SIZE = 768
NUM_LAYERS = 16 # deeper + wider
MAX_LENGTH = 128 # 2x context
DROPOUT = 0.05 # less dropout for bigger model
BATCH_SIZE = 4 # smaller batch (more memory per sample)
LEARNING_RATE = 0.0005 # lower LR for bigger model

# === Aether-Large — ~350M params, needs GPU ===
HIDDEN_SIZE = 1024
NUM_LAYERS = 24
MAX_LENGTH = 512 # long context
DROPOUT = 0.1
BATCH_SIZE = 32
LEARNING_RATE = 0.0003 # standard for ~350M

2. Dataset scaling in generate_dataset.py

Bigger models need proportionally more diverse data. Each existing section has a multiplier pattern. To scale data 5x:

# Multiply every section's output
def generate(self):
    for _ in range(5): # <-- repeat loop
        self.gen_identity()
        self.gen_greetings()
        self.gen_knowledge()
        self.gen_cs_hardware()
        # ... all other sections ...
        self.gen_casual_multi()

3. Scaling rules of thumb

Data ≈ 20x params

For 100M params aim for ~2B tokens. The current 580K tokens is enough for 24.7M, but a 100M model will quickly overfit on it.

Lower LR for bigger models

Halve the learning rate when you double the width. The 0.001 used for 24.7M is too aggressive for 100M+. Stick to AdamW with selective weight decay.

Dropout shrinks as data grows

With 580K tokens we needed dropout=0.1. With 50M+ tokens, dropout=0.05 or even 0.0 works better. Over-regularization limits capacity.

Context length trade-off

Longer MAX_LENGTH improves coherence but quadratically increases memory. For CPU, stick to 64-128. For GPU, try 512-1024.

Use the official RWKV recipe

For 100M+ models, follow BlinkDL's RWKV-LM init gains and lr-per-parameter. The values we use match their small-model config.

Watch your checkpoint size

A 350M model is ~1.4GB. Make sure your disk has room for several checkpoints and your RAM is ≥ 32GB for CPU training at that scale.
